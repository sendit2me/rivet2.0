# RivetDebuggerServer

## Description

`RivetDebuggerServer` is the server object returned by [startDebuggerServer](./startDebuggerServer). It owns the remote-debugger WebSocket server and can attach graph processors so the Rivet app can inspect, pause, resume, abort, and stream execution state.

Pass a debugger server to `runGraph` or `createProcessor` with the `remoteDebugger` option when you want a Node execution to appear in Rivet's remote debugger.

## Definition

```typescript
export interface RivetDebuggerServer {
  on: Emittery<DebuggerEvents>['on'];
  off: Emittery<DebuggerEvents>['off'];

  webSocketServer: WebSocketServer;

  broadcast(
    processor: GraphProcessor,
    message: string,
    data: unknown,
    requestId?: RemoteRunRequestId,
  ): void;

  attach(processor: GraphProcessor, requestId?: RemoteRunRequestId): void;
  detach(processor: GraphProcessor): void;
}
```

## Properties

### on

Type: `Emittery<DebuggerEvents>['on']`

Subscribes to debugger-server events.

### off

Type: `Emittery<DebuggerEvents>['off']`

Unsubscribes from debugger-server events.

### webSocketServer

Type: `WebSocketServer`

The underlying `ws` server instance.

## Methods

### broadcast(processor, message, data, requestId?)

Sends a debugger message to the clients associated with a processor.

Most integrations do not call this directly. The debugger server wires processor events to broadcasts when `attach` is called.
Server-to-client debugger sends, including broadcasts and connection-time messages, are best-effort: serialization or websocket
send failures are emitted through the debugger `error` event, the failed websocket is terminated, and graph execution continues.

### attach(processor, requestId?)

Attaches a `GraphProcessor` to the debugger server. Attached processors broadcast execution events to Rivet clients.

The Node package calls this for you when `remoteDebugger` is passed to `createProcessor` or `runGraph`; `createProcessor`
also re-attaches before each `run()` so reused processor objects continue to emit debugger events.
Processor attachments are run-scoped: the debugger server removes its processor listeners when `detach` is called and also
auto-detaches after the processor emits its root `finish` event.

Processors created through `createProcessor(..., { remoteDebugger })` include `durationMs` on `nodeFinish` and `nodeError`
debugger events by default, and split-run nodes include `splitRunDurationMs` keyed by split item index. Pass
`captureNodeTimings: false` to `createProcessor` if the remote backend should suppress that optional metadata.

### detach(processor)

Detaches a processor from the debugger server and removes the debugger event listeners registered by `attach`. Calling
`detach` more than once for the same processor is safe.

## Usage

```typescript
import { loadProjectFromFile, runGraph, startDebuggerServer } from '@valerypopoff/rivet2-node';

const project = await loadProjectFromFile('./workflow.rivet-project');
const remoteDebugger = startDebuggerServer({ port: 21888 });

remoteDebugger.on('error', (error) => {
  console.error(error);
});

await runGraph(project, {
  graph: 'Main',
  remoteDebugger,
});
```

## Notes

- The default debugger port is `21888`.
- `startDebuggerServer` accepts a `host` option. Use an explicit host when exposing the debugger outside localhost.
- `startDebuggerServer` sends WebSocket ping frames by default so idle remote-debugger sockets stay alive through common proxy and CDN idle timeouts. Real debugger traffic also resets any pending heartbeat timeout.
- Server transport sends and processor attachment cleanup are best-effort server policies; they do not change the websocket message schema or require app-side reconnect behavior for external debugger sessions.
- The app executor uses a separate internal executor WebSocket contract. `RivetDebuggerServer` is for programmatic Node executions that want remote-debugger inspection.
