# createGraphRunner

## Description

`createGraphRunner` creates a reusable Node runner for headless or backend integrations that load a project once and run the same graph many times.

Use it when you do not need Remote Debugger attachment, recording playback, or event streams for each run. Those surfaces should keep using `createProcessor`, `runGraph`, or `runGraphInFile`.

The runner keeps stable setup at creation time:

- graph selection
- registry and plugin setup
- Node defaults such as `NodeNativeApi`, `NodeMCPProvider`, the Node CodeRunner, `NodeProjectReferenceLoader`, and the fallback tokenizer
- process settings and plugin environment values
- project path and project-reference loader
- event callbacks supplied in the creation options

Each `run` call owns the values that naturally vary per request:

- graph inputs
- graph context values
- abort signal

Each `run` call uses a run-scoped processor so mutable graph state, including Global node values, does not leak between requests. The runner still reuses the stable Node runtime context instead of reconstructing provider defaults for every call.

## Syntax

```typescript
function createGraphRunner(project: Project, options: NodeGraphRunnerOptions): NodeGraphRunner;

type NodeGraphRunnerOptions = Omit<
  NodeRunGraphOptions,
  'abortSignal' | 'context' | 'inputs' | 'remoteDebugger' | 'remoteDebuggerRequestId'
> & {
  runtimeProfile?: 'compatible' | 'headless-fast';
};

type NodeGraphRunnerRunOptions = {
  inputs?: Record<string, LooseDataValue>;
  context?: Record<string, LooseDataValue>;
  abortSignal?: AbortSignal;
};

type NodeGraphRunner = {
  run(options?: NodeGraphRunnerRunOptions): Promise<Record<string, DataValue>>;
  dispose(): void;
};
```

`runtimeProfile` controls runner-only optimizations:

- `compatible` uses the same default `NodeCodeRunner` behavior as `runGraph` and
  `createProcessor`.
- `headless-fast` still uses the compatible `GraphProcessor` execution path, but
  it reuses an immutable graph execution plan across runs and, when you do not
  pass a custom `codeRunner`, uses a runner-owned cached Node CodeRunner for
  Code-family nodes.

The fast profile caches graph topology, validated connections, port definitions,
planner adjacency maps, and loaded project-reference snapshots for the runner's
stable project/registry setup. Plans are keyed per graph, so nested subgraphs
and referenced graphs can reuse their own planning data too. It does not cache
output values, input values, graph inputs, context values, or node
implementation instances, so each `run` call still receives fresh request data,
fresh node implementations, and a run-scoped processor.

## Parameters

- `project` (Project): The loaded Rivet project.
- `options` (NodeGraphRunnerOptions): Stable runner configuration. `inputs`, `context`, and `abortSignal` are intentionally not creation-time options. `remoteDebugger` is not supported by this runner path.

## Return

Returns a `NodeGraphRunner`. Call `run` for each request and `dispose` when the runner should no longer accept new runs.

## Example

```typescript
import { createGraphRunner, loadProjectFromFile } from '@valerypopoff/rivet2-node';

const project = await loadProjectFromFile('./workflow.rivet-project');
const runner = createGraphRunner(project, {
  graph: 'Main',
  runtimeProfile: 'headless-fast',
});

const outputs = await runner.run({
  inputs: {
    prompt: 'Summarize the release notes',
  },
  context: {
    audience: 'support team',
  },
});

console.log(outputs);
```

## When To Use It

Use `createGraphRunner` for repeated backend or API-server calls where the project and graph are stable and only inputs/context change.

Keep using:

- `runGraphInFile` for simple one-shot scripts.
- `runGraph` when you already loaded a project but still want one-shot execution.
- `createProcessor` when you need processor events, SSE streams, node streams, recording, or Remote Debugger behavior.
